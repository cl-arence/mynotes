# Mynotes Learning Context

- This is a Flutter course project based on the approximately 37-hour freeCodeCamp Flutter video on YouTube: https://youtu.be/VPvVD8t02U8
- The learner was at roughly the 18-hour mark on June 20, 2026.
- The purpose of this project is to learn Flutter and prepare to build original apps in Visual Studio Code.
- Git reference for reusable login and Firebase authentication code: commit `dd40333` (`step-10`) is the last commit before SQLite/local CRUD functionality was introduced. Commit `6a7a0fb` (`step-11`) begins the SQLite work.
- Git reference for the completed local SQLite CRUD phase: commit `c986fbc` (`step-18`) is the clean checkpoint immediately before the course begins moving notes from local CRUD storage to Firebase. Preserve this reference for future explanations and reuse of the CRUD architecture.
- As of after `step-18`, the course is transitioning from local SQLite CRUD notes to Firebase-backed notes. The learner expects to revisit the CRUD code later and may need a thorough conceptual explanation of how its database, cache, streams, models, and views work together.
- Key Firebase transition noted by the learner: the Firestore implementation no longer needs the manually managed notes `StreamController` used by the SQLite CRUD service. Firestore supplies real-time data streams directly through snapshot listeners, reducing the need for a separate local cache-and-broadcast layer.
- Next planned learning task as of June 22, 2026: begin the next session with an explanation of the Firebase cloud-storage code introduced in `step-19`, especially `firebase_cloud_storage.dart`, `cloud_note.dart`, the storage constants, and the cloud-storage exceptions.
- Git reference `abc9192` (`step-22`) is a deliberate course deviation/detour to test BLoC by implementing a counter. Treat it as a learning checkpoint for BLoC concepts rather than the main notes-app architecture.
- Git reference `6a80179` (`step-27`) is the checkpoint where the app is more or less migrated to BLoC for authentication flow. It includes auth BLoC/events/states, login/register BLoC refactors, loading screen handling, forgot-password support, and related auth test updates.
- Release workflow note: the learner develops on Windows and does not want to buy a Mac. For course sections involving iOS release, Xcode, storyboards, launch screens, signing, TestFlight, or App Store upload, translate the Mac/Xcode instructions into Windows-friendly options when possible. Prefer Flutter-native packages such as `flutter_launcher_icons` and `flutter_native_splash` for icons/splash screens. For future iOS release needs, revisit options such as Codemagic, Bitrise, GitHub Actions macOS runners, Appcircle, MacStadium, or MacinCloud. Recommended path for now is Windows-first development, Android release from Windows, then cloud Mac tooling only when iOS publishing is worth pursuing.
- iOS/cloud-Mac workflow to revisit: on Windows, the learner can prepare Flutter/Dart code, assets, Android builds, and Git history, but cannot complete the normal iOS build/sign/upload flow locally. For iOS, plan to push/link the full Git repo to a cloud Mac service. The service checks out the repo on macOS, runs Flutter/CocoaPods/Xcode build commands, signs the app, creates the archive/IPA, and uploads to App Store Connect/TestFlight. Revisit paid Apple Developer enrollment, bundle ID, certificates, provisioning profiles, and App Store Connect API keys/credentials when ready. Codemagic is likely the first service to evaluate because it is Flutter-focused and beginner-friendly.
- Future production hosting note: this course used GitHub Pages from the public `mynotes` repo for privacy-policy and account-deletion URLs. For future real apps, avoid requiring private source repos to be public just to host policy pages. Revisit alternatives such as a separate public policy-only repo, GitHub Pages from a minimal docs repo, Netlify, Vercel, Firebase Hosting, Cloudflare Pages, or a simple company/app website. Keep source code private while publishing only the required public legal/support pages.
- Future-app workflow: the learner plans to use AI as a CTO/coding partner while still understanding and debugging the code. When starting future Flutter apps, prefer a structure similar to this `mynotes` project so it remains familiar: `lib/views` for screens, `lib/services/auth` for authentication abstractions and Firebase auth provider, `lib/services/auth/bloc` for auth BLoC/events/states, `lib/services/cloud` for Firestore-backed app data, `lib/constants` for routes and field names, `lib/utilities/dialogs` or `lib/helpers` for reusable UI helpers, and tests where practical.
- For future apps, be ready to scaffold generic reusable starter code first: Firebase initialization, login/register/logout/email verification/forgot-password flow, auth provider abstraction, auth BLoC, loading/error dialogs, routes, basic app shell, and Firestore service/model patterns. Then build app-specific functionality on top.
- If the learner asks to build an entire app from a rough concept, default to a `mynotes`-like architecture unless the app clearly needs something different. Explain the structure enough that the learner can follow modifications and debugging later; do not create an opaque or over-engineered architecture.
